# 迭代回顾报告

## Step 1: 获取迭代信息

```
读取 .yunxiao.json 获取 organizationId、spaceId

确定目标迭代：
  方式一：用户指定迭代名称 → list_sprints 匹配
  方式二：自动选取最近完成的迭代 → list_sprints(status: ["ARCHIVED"])，取最新一个
  方式三：当前进行中迭代 → list_sprints(status: ["DOING"])

调用 get_sprint 获取迭代详情：起止日期、负责人、容量
```

## Step 2: 采集迭代工作项

```
调用 search_workitems：
  - sprint: 迭代 ID
  - category: "Req,Task,Bug"
  - perPage: 200

分类统计：
  - 计划内 vs 计划外（紧急插入）
  - 按 category 分组：需求/任务/缺陷
  - 按 status 分组：已完成/进行中/未开始
  - 按 assignedTo 分组：每人贡献
```

## Step 3: 分析与评估

```
核心指标：
  1. 完成率 = 已完成 / 总计划 × 100%
  2. 需求交付率 = 已完成需求 / 计划需求
  3. 缺陷解决率 = 已关闭 Bug / 总 Bug
  4. 延期项数量及原因分析

延期分析：
  筛选 planEndDate < 迭代结束日期 且未完成的工作项
  尝试归因：
    - 需求变更
    - 技术复杂度低估
    - 外部依赖阻塞
    - 人力不足

紧急插入分析：
  识别迭代开始后创建的工作项
  统计对计划的影响
```

## Step 4: 生成报告

```

> 周期：YYYY-MM-DD ~ YYYY-MM-DD

## 📊 完成情况
| 类型 | 计划 | 完成 | 完成率 |
|------|------|------|--------|
| 需求 | X | X | XX% |
| 任务 | X | X | XX% |
| 缺陷 | X | X | XX% |
| **合计** | X | X | XX% |

## 👥 成员贡献
| 成员 | 完成数 | 延期数 | 完成率 |

## ⚠️ 延期项分析
| 编号 | 标题 | 负责人 | 延期天数 | 可能原因 |

## 🔥 紧急插入项
| 编号 | 标题 | 插入日期 | 当前状态 |

## 💡 改进建议
- 基于数据自动生成 3-5 条改进建议
- 如：需求评审阶段增加技术评估、控制迭代中途需求变更
```
